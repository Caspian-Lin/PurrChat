# Bot Studio 特殊模式节点图 — 改进方案

> 基于 n8n、Dify、LangGraph Studio、ComfyUI 等开源项目的调研，结合 PurrChat "低门槛 Agent 编排"定位设计的系统性改进方案。

## 一、当前实现诊断

### 1. 循环节点 — 核心问题

- **「循环体入口」是假的**：纯视觉 `div`，不是 Vue Flow Handle，无法连线
- **容器 `pointer-events: none`**：整个框体无法交互
- **循环体子节点的 "回边" 被过滤**：用户看不到循环体的终止和返回
- **缺少循环迭代变量**：没有 `index`/`item` 自动注入

### 2. 条件判断节点 — 功能不足

- 只支持单个二元比较，无 AND/OR 逻辑组合
- 只有二分支（true/false），无多路 Switch
- 端口命名不友好（"左操作数"/"右操作数"）

### 3. 变量系统 — 认知负担过高

- 变量引用语法 `$nodeID:portID` 是机器友好的，用户必须记住 UUID
- 没有任何 UI 辅助来发现可用变量
- 没有类型信息和自动补全

### 4. 缺失能力

| 能力 | 当前状态 | 对比参考 |
|------|---------|---------|
| 变量选择器（VarReferencePicker） | 无 | Dify |
| 循环迭代变量（index/item） | 无 | Dify |
| 复合条件（AND/OR） | 无 | n8n |
| 多分支 Switch | 无 | n8n |
| 图 ↔ 代码双向转换 | 仅 JSON 导入导出 | LangGraph Studio |
| 执行调试（逐步查看状态） | 无 | LangGraph Studio |
| 工具调用节点（HTTP/API） | 无 | n8n |
| 错误处理分支 | 无 | Dify |

## 二、改进方案设计

### 层次 1：修复可用性问题（短期）

#### 1.1 循环节点重构 — 从"容器"变为"回环"

放弃嵌套容器模式，改用 **n8n 风格的回环（loopback）模式**：

```
修改前（容器模式）：
┌─── loop ──────────────────┐
│  ▶执行              完成▶  │
│  ·条件    ▶循环体入口     │
│      [子节点们]           │
└───────────────────────────┘

修改后（回环模式）：
[loop] ──out_body──→ [处理A] ──→ [处理B] ──┐
                                        │
←────────────────────────────────────────┘

[loop] ──out_done──→ [后续节点]
```

- Loop 节点退化为普通大小的节点，在扁平画布上与其它节点并列
- `out_body` 连接循环体第一个节点，最后一个节点回连到 loop 的 `in_exec`
- 回环线可见，用户一眼看出"这是一个循环"
- 自动注入 `loop.index`（迭代序号）和 `loop.iterations`（总次数）变量

#### 1.2 If 节点升级 — 支持多条件 + 标签

- 增加多条件列表（可添加/删除条件）
- 条件之间支持 AND/OR 切换
- 端口标签支持自定义命名

#### 1.3 变量选择器组件（VarReferencePicker）

参考 Dify 实现：
1. 遍历上游节点（通过 connections 反向追踪），收集输出端口
2. 按"节点 → 端口"两级结构展示，每个端口附带类型标签
3. 用户点选后插入 `{nodeLabel.portName}` 格式引用
4. 后端 `replaceVariables` 同步支持人类可读格式

### 层次 2：增强 Agent 编排能力（中期）

#### 2.1 Switch 节点（多分支路由）
- 一个输入 + N 个可配置输出端口 + 默认端口
- 每个端口可配置匹配值或表达式

#### 2.2 Merge 节点（分支汇聚）
- 允许多条分支汇聚到同一个后续节点

#### 2.3 Tool 节点
- HTTP 请求节点（GET/POST，JSON body）
- 增强现有的 Python 节点为更通用的 Code 节点

### 层次 3：代码 ↔ 图双向转换（长期）

#### 3.1 IR 中间表示层（YAML）

设计人类可读的 YAML 格式，作为图和代码的双向桥梁。技术用户和 Claude Code 可直接编辑 YAML，非技术用户可拖拽节点，两者编辑同一份源文件。

#### 3.2 Claude Code 集成
- Claude Code 直接编辑 YAML 文件来修改 Agent 逻辑
- Bot Studio 自动检测 YAML 变化，更新可视化图
- YAML 可 git diff、code review、版本管理

## 三、开源方案对比

| 方案 | 循环设计 | 条件分支 | 变量系统 | 图↔代码 | 借鉴要点 |
|------|---------|---------|---------|---------|---------|
| n8n | 回环连线 | IF + Switch + 多条件 | 表达式 + 拖拽引用 | AI生成 + JSON | 循环回环、Switch |
| Dify | 容器嵌套子画布 | IF + 条件组合 | VarReferencePicker | 无 | 变量选择器、迭代变量 |
| LangGraph | Python for循环 | 条件边 | Python 变量 | 代码→图投影 | 代码优先哲学、调试 |
| ComfyUI | 自定义Loop | If节点 | 纯连线传播 | 无 | 端口数据流 |

**组合策略**：n8n 循环回环 + Dify 变量选择器 + LangGraph 代码↔图理念

## 四、实施优先级

| 优先级 | 改进项 | 预期效果 |
|--------|--------|---------|
| **P0** | 循环节点改为回环模式 | 修复"完全不可用" |
| **P0** | VarReferencePicker 组件 | 用户能发现和使用变量 |
| **P1** | If 节点 AND/OR + 多条件 | 表达更复杂分支逻辑 |
| **P1** | 变量语法改为 `{nodeLabel.portName}` | 大幅降低认知负担 |
| **P1** | 循环注入 index/iterations 变量 | 循环体可做迭代逻辑 |
| **P2** | Switch 多分支节点 | N 路路由 |
| **P2** | Merge 汇聚节点 | 分支合流 |
| **P2** | YAML 中间表示层 | 代码↔图基础 |
| **P3** | 图 ↔ YAML 双向同步 | Claude Code 可编辑 Bot |
| **P3** | Tool 节点（HTTP/代码执行） | Agent 调用外部能力 |
